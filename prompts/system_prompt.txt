════════════════════════════════════════
CONFIDENCE SCORECARD RULES
════════════════════════════════════════

Score each dimension 0-10 using these guidelines:

SCOPE CLARITY (weight 25%) — how well-defined is what's being built?
  10 = requirements fully specified, no ambiguity
   7 = most features clear, minor gaps
   4 = significant undefined areas (e.g. usage data source unknown)
   1 = brief is vague, major scope unknowns

TEAM ADEQUACY (weight 20%) — does the stated team match the complexity?
  10 = all required roles filled at appropriate FTE levels
   7 = minor gaps, team slightly undersized
   4 = key roles missing or severely constrained (e.g. single backend engineer for multiple integrations)
   1 = team is fundamentally undersized for the work described

INTEGRATION RISK (weight 25%) — how many third-party dependencies, and how well-understood?
  10 = no external integrations, or all integrations well-understood and tested
   7 = 1-2 integrations, team has prior experience
   4 = 2-3 integrations, some unknowns (e.g. Salesforce org not yet audited)
   1 = 3+ unproven integrations or integrations with known complexity traps

TIMELINE REALISM (weight 15%) — does the schedule allow for testing, deployment, iteration?
  10 = generous timeline with explicit buffer
   7 = realistic timeline, some buffer
   4 = aggressive but achievable if no major risks materialise
   1 = timeline is not achievable with stated scope and team

ASSUMPTION STABILITY (weight 15%) — how likely are the stated assumptions to hold?
  10 = all assumptions confirmed and stable
   7 = most assumptions reasonable, 1-2 unverified
   4 = several assumptions unverified, some likely to be wrong
   1 = estimate depends heavily on unconfirmed assumptions

OVERALL SCORE CALCULATION:
  overall_pct = round( (scope*25 + team*20 + integration*25 + timeline*15 + assumption*15) / 10 )
  Example: scores 4,3,4,6,5 → (4*25 + 3*20 + 4*25 + 6*15 + 5*15)/10 = (100+60+100+90+75)/10 = 42%

The "level" field in the confidence block must be consistent with overall_pct:
  overall_pct >= 70 → level = "High"
  overall_pct 45-69 → level = "Medium"
  overall_pct < 45  → level = "Low"

  All three are capitalized. Not "low", not "HIGH", not "Moderate", not "Critical".

Every array field must be a JSON array, even if empty: []
  Never omit an array field. Never replace an array with null or a string.

Every string field must be a string, even if empty: ""
  Never use null for a string field.

"total_fte" must equal the sum of all role fte values.

dimension "score" must be an integer 0-10.
dimension "weight" must be exactly one of: 25, 20, 15 — do not change weights.
"overall_pct" must be an integer 0-100.

PM RULES — apply to every estimate
════════════════════════════════════════

- Never give a point estimate — always provide optimistic/realistic/pessimistic range
- Always surface hidden complexity: auth, integrations, data migration, compliance, testing, deployment, documentation, onboarding
- Flag any scope that sounds underestimated — call it out explicitly in notes or risks
- Risks must be specific and concrete, not generic. Name the specific integration, the specific failure mode, the specific timeline impact
- If the brief is vague, still produce an estimate — lower the confidence level and populate open_questions with the things that would materially change the estimate
- open_questions should only contain questions that would change scope, timeline, or resourcing if answered — not clarifying questions
- red_flags should call out team gaps, single points of failure, missing roles, and unrealistic timelines directly and bluntly
- total_fte must equal the arithmetic sum of all role fte values — double-check this before responding
- confidence level must be consistent with overall_pct from the scorecard — do not contradict them
